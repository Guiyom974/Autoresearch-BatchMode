Based on the research problem and the constraints provided, here are four testable hypotheses designed to validate the transition to an in-process architecture.

### Hypothesis 1: Elimination of Path-Related Execution Failures
**Hypothesis Statement:** Migrating from `subprocess` invocations to direct function calls within a `tempfile.TemporaryDirectory()` context will result in a 0% failure rate for `FileNotFoundError` and `PermissionError` exceptions during execution.

**Why it’s testable:** This hypothesis is binary and falsifiable. By running the pipeline through a controlled stress test (e.g., 50 consecutive iterations), we can programmatically track and log any raised OS exceptions. If even one exception occurs, the hypothesis is rejected.

**Experiment:** Implement a wrapper that executes the prime generation and artifact saving logic in a loop 50 times. Use a `try-except` block specifically catching `FileNotFoundError` and `PermissionError`. The success metric is the total absence of these specific exceptions across all 50 iterations, compared to a baseline of the previous `subprocess` implementation which previously encountered these errors.

---

### Hypothesis 2: Artifact Integrity via In-Memory Verification
**Hypothesis Statement:** By capturing file handles directly from the in-process generation functions, the system can guarantee 100% artifact integrity (defined as file existence and non-zero byte size) before the context manager closes the temporary directory.

**Why it’s testable:** This hypothesis rests on the ability to programmatically inspect the file system state *before* cleanup. It is testable because file metadata (size, existence) is easily queried via the `os` and `pathlib` modules in Python.

**Experiment:** After the analysis function completes but *before* the `with` block for `TemporaryDirectory()` exits, include an assertion suite that checks for the existence of the required output files (e.g., `primes.csv`, `gap_histogram.png`). Verify that `os.path.getsize(file_path) > 0` for every required artifact. If the assertion fails, the hypothesis is rejected.

---

### Hypothesis 3: Statistical Accuracy of Unified Pipeline
**Hypothesis Statement:** The prime gap distribution generated by the in-process architecture for primes up to 10^5 will deviate from mathematically established expectations (e.g., the Prime Number Theorem) by less than 0.1%.

**Why it’s testable:** Prime gap statistics are deterministic. We can calculate the expected mean gap (approximately $\ln(n)$) and frequency distributions using a trusted, independent reference implementation and compare them against the outputs of our new in-process pipeline.

**Experiment:** Create a "Golden Dataset" using a standard, well-tested Python library or a simple, verified Sieve implementation outside the pipeline. Run the new in-process pipeline to generate the same data. Compare the mean prime gap and the count of specific residues (e.g., mod 6) between the two datasets. If the variance between the two exceeds the 0.1% threshold, the hypothesis is rejected.

---

### Hypothesis 4: Execution Efficiency under Temporal Constraints
**Hypothesis Statement:** The in-process architecture will reduce total execution time (setup, generation, analysis, plotting, and cleanup) to under 15 seconds for primes up to 10^5, significantly outperforming the 2-minute upper limit and the previous subprocess-based execution time.

**Why it’s testable:** This is measurable using Python's `time` module. It tests the overhead reduction gained by eliminating the creation of new OS processes and the overhead of serializing data between processes.

**Experiment:** Use the `time.perf_counter()` function to measure the exact duration of the entire block of code inside the `TemporaryDirectory` context. Perform 10 runs to calculate the mean and standard deviation of execution time. If the mean execution time exceeds 15 seconds, the hypothesis is rejected (even if it stays under the 2-minute limit, as the hypothesis specifically predicts a significant efficiency gain).
